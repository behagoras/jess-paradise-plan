Red-Team Critique 1

The product says "Don't pick where. Pick the feeling", but the implemented experience still behaves like a long form before discovery. This undercuts the core thesis.

Red-Team Critique 2

The app asks for many inputs that do not influence matching. This makes the UI feel slower and can break trust once the maintainer notices the scoring code.

Red-Team Critique 3

The docs describe three products at once: original 2019 UI, current Next prototype, and target Drift/feed MVP. They need explicit labels or future work will drift.

What Would Break Trust?

Booking-adjacent claims without booking behavior

Provider handoff copy is currently ahead of implementation. In travel, this is more damaging than ordinary UI polish debt because users expect prices, availability, and provider links to be precise.

Next action: build real click-out or make the CTA explicitly prototype-only.

Artificial scarcity and savings

"Seats left", percentage discounts, ratings, "matched this week", and scanning copy are all powerful persuasion mechanisms. Until backed by data, they should read as demo content.

Next action: replace with "sample data" in dev or add real observed_at/source fields.

Hidden mismatch between effort and outcome

Asking for filters that do not affect results teaches the user to distrust personalization.

Next action: every visible input must affect ranking, detail, or saved-filter behavior.

Docs that do not tell maintainers what is real

The docs are valuable, but they mix original design, future strategy, and current implementation. A maintainer could build the wrong next thing from the wrong source.

Next action: add current/target labels and a "source of truth by task" section.
