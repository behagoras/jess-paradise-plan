1Red-team summary

Five adversarial passes, all returning high confidence. The strongest critique is singular: the product collects far more than it uses and signals more than it has — a long wizard whose effort is largely decorative, wrapped in fabricated live-data cues, documented as a different product entirely.

UI Reality Checkconfidence: high

Step 1 is 1,111px tall on a 844px iPhone viewport and collects 33 interactive inputs, but 5 of its 7 named fields are dead — they are stored in state but never read by the scoring or pricing logic. Step 3 adds a full redundant summary screen and tap. The Activity "Swap" button in the Detail screen has no onClick handler and does nothing. Hardcoded dates and scarcity counts in data.ts contradict the "When" selection the user just made, undermining trust in the results.

T2confidence: high

The README.md "Architecture" and "Key files" sections are deeply stale — they predate the move to src/, the shadcn/ui + Tailwind v4 re-platform, and the removal of PhoneFrame/StatusBar/theme.ts. Beyond the README, the docs/wiki/ vault (1,700+ lines) documents an unbuilt Drift feed product with Amadeus/Travelpayouts integration, while every actual source file is a hardcoded 6-destination wizard with no live data. docs/flujo-app-ui.md documents a third incompatible product (global nav + Checkout). There is zero documentation that accurately describes the real built app for a solo maintainer.

UX / Docs Mismatchconfidence: high

The repo contains three incompatible mental models that are never reconciled in any document: (1) the 2019 XD flow that ends with a first-party "Checkout" / "Reserva confirmada"; (2) the built wizard (English, 9 fields on one step, Convex persistence on handoff, no live data); (3) the Drift discovery-first vision (Spanish, feed + "Sorpréndeme", affiliate-only redirect, days-available filter, Amadeus/Travelpayouts). The built app partially satisfies (2) and surfaces signals of (3) in copy ("flexibility unlocks the best prices", "from estimates") but structurally contradicts both (1) and (3) in entry flow, language, CTA, and scope. No single document describes what was actually shipped; the Drift wiki describes a product that does not exist, and the Mapa-de-Pantallas / flujo-app-ui docs describe a product that was explicitly retired.

Adversarial First-Time Userconfidence: high

A skeptical first-time user faces 8 mandatory decision groups before seeing any payoff, collapses that work onto a single scrolling screen, then arrives at results that carry fake scarcity signals ("4 seats left", "Departs in 18 days"), a discount that is purely computed from a formula rather than real data, and a "Swap" activity button that silently does nothing. The product's own tagline ("60s to a full trip") is falsified by the wizard's length. Five fields collected in Step 1 (departure, budget, packages, vacation type, traveler type) are never wired into the scoring algorithm, which only reads general/weather/intensity — meaning the "personalization" promise is structurally broken for the majority of user input.

T5confidence: high

usePlanner.ts is a genuine god object: it owns state type, initial values, all navigation transitions, the animated loader timers, destination scoring, AND all pricing/discount math in a single 262-line hook with 15 graph edges. Five of the eight PlannerState fields collected by the wizard (budget, departure, vacation, travelerTypes, packages) are never read by the scoring or pricing logic — they are display-only. Pricing is driven entirely by nine undocumented magic-number multipliers with no comment tying them to source data. The README points to three files that do not exist (theme.ts, PhoneFrame.tsx, StatusBar.tsx), lists paths without the src/ prefix, and calls the styling layer "CSS custom properties + inline styles" when the app is Tailwind v4 + shadcn/ui. A Detail screen "Swap" EditButton has no onClick handler. These issues collectively guarantee a wrong mental model on day one.

6What would confuse a maintainer?

What the solo owner — or a new collaborator — would get wrong returning to this code next week.

highP1large

usePlanner is a god object: state machine + navigation + timers + scoring + pricing in one hook

ProblemusePlanner.ts (262 lines, 15 graph edges) owns: the Screen state machine and PlannerState type (lines 11-57); all navigation transitions (startWizard, goStep, prevStep, nextStep, go — lines 114-152); the three hardcoded animation timers for the generating screen (950 / 1850 / 2900 ms, lines 140-145); destination scoring via computeOptions (lines 155-166); and the entire pricing formula with all multipliers and discount logic in a 76-line useMemo (lines 168-244). The Graphify god-node report confirms usePlanner() at 15 edges, meaning every screen component is directly coupled to its shape via PlannerApi.
Why it mattersAny maintainer wanting to change just one concern — e.g., adjust the scoring algorithm, swap the animation timing, or add a new screen — must navigate all of the above at once. Community 0 cohesion of 0.06 (screen components weakly interconnected) is a symptom: the components don't reference each other because they all go through this single hub. Adding a 7th destination, changing the discount model, or replacing the timed loader with a real async call all require editing the same file.
Proposed changeBefore: all concerns in usePlanner.ts. After: extract (1) a pure computeOptions(general, weather, intensity) function + pure computePricing(dest, state) function into src/lib/scoring.ts and src/lib/pricing.ts — these have zero React dependencies; (2) move the timer constants (PHASE_DELAYS = [950, 1850, 2900]) into src/components/screens/Generating.tsx or a co-located constant since they are purely a UX animation concern; (3) keep usePlanner as a thin coordinator that calls the extracted functions. PlannerApi shape stays unchanged so screens need no edits.
Tradeoff / residual riskExtraction adds files. If the app never grows beyond 6 destinations the current coupling is low friction. The real cost is deferred until the first time scoring or pricing needs to change independently.
