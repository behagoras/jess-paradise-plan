01

Overview

From ideas to a build plan

Five tracks (A smart-filters/slim-wizard, B favorites, C saved-filters+alerts+email, D StumbleUpon discovery, E AI "¿y qué hago ahí?" curation) all hang off two facts about the current code: (1) scoring lives only client-side in usePlanner.ts:155-166 and ignores everything except general/weather/intensity, and (2) persistence is one `trips` table with the userId-OR-guestId + claim-on-sign-in pattern (convex/schema.ts:11-26, convex/trips.ts). The synthesis is: make scoring REAL and SHARED first (Track A + extract a pure scoreDestinations), because favorites' deterministic re-open (B), the alert matcher (C), and the discovery shuffle (D) all consume that same scoring contract. Persist every new user object through ONE owner-pattern (userId?/guestId? + by_user/by_guest + claim-on-sign-in) reusing the exact shape already proven by trips. Stand up shared infra (Convex crons, Resend via fetch, a notifications channel, AI Gateway) once and let C/D/E draw on it. Ship a tiny first PR (shared constants + exported tripFields + pure scoring module) that unblocks all four data tracks without UI risk. Defer AI/discovery polish until the real-scoring + favorites loop is closed, matching the "experience feels long" concern — the wizard slim (A) and favorites (B) are what actually shorten the funnel today.

Your ideas → these tracks

  • "Days you wanna go" + optional filters → Track A (smart filters)
  • Favorite trips → Track B deepened by Opus
  • Saved filters · new-trip notifications · destination emails → Track C
  • StumbleUpon-style discovery → Track D
  • Research AI experience → Track E (conversational "Sorpréndeme")

Unified data model

One coherent Convex schema

All five ideas reuse the proven userId? XOR guestId? owner pattern with claim-on-sign-in. Favorites, saved filters, notifications, interactions, curations — designed together, not as four conflicting schemas.

// convex/schema.ts — ONE coherent design. Every new table copies the trips
// owner pattern: userId?:string (Clerk subject) XOR guestId?:string (pp_guest),
// .index("by_user",["userId"]) + .index("by_guest",["guestId"]), and a
// claim-on-sign-in mutation that patches userId / clears guestId.
//
// tripFields is MOVED out of convex/trips.ts (currently the module-local const
// at trips.ts:5-16) to convex/lib/tripFields.ts and imported by trips.ts +
// favorites.ts so favorite/trip payloads never drift.

import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";
import { tripFields } from "./lib/tripFields"; // moved from trips.ts:5

export default defineSchema({
  // EXISTING — unchanged shape, now sources tripFields from the shared module.
  trips: defineTable({
    userId: v.optional(v.string()),
    guestId: v.optional(v.string()),
    ...tripFields,
  })
    .index("by_user", ["userId"])
    .index("by_guest", ["guestId"]),

  // TRACK B — favorites. Separate table (NOT a status on trips) because
  // saveDraft enforces one-draft-per-guest via .first() upsert (trips.ts:36-44);
  // favorites are many-per-owner. Adds airlineIdx/hotelIdx (NOT in tripFields)
  // so re-open reproduces the exact priced Detail (usePlanner.ts:184-191).
  favorites: defineTable({
    userId: v.optional(v.string()),
    guestId: v.optional(v.string()),
    ...tripFields,
    airlineIdx: v.number(),
    hotelIdx: v.number(),
  })
    .index("by_user", ["userId"])
    .index("by_guest", ["guestId"])
    .index("by_user_dest", ["userId", "destinationKey"])   // isFavorited + toggle dedupe
    .index("by_guest_dest", ["guestId", "destinationKey"]),

  // TRACK C — saved searches. Criteria = the subset of PlannerState that the
  // shared scoreDestinations() reads, plus Corte-1-forward optional fields.
  savedFilters: defineTable({
    userId: v.optional(v.string()),
    guestId: v.optional(v.string()),
    name: v.string(),
    general: v.array(v.string()),
    weather: v.array(v.string()),
    intensity: v.number(),
    daysMax: v.optional(v.number()),   // ties to Track A daysAvailable
    budget: v.optional(v.number()),
    departure: v.optional(v.string()),
    createdAt: v.number(),
    lastNotifiedAt: v.optional(v.number()),
    lastMatchKeys: v.optional(v.array(v.string())), // delta detection vs new run
  })
    .index("by_user", ["userId"])
    .index("by_guest", ["guestId"]),

  // TRACK C+D — ONE notifications channel feeds both alert results and (later)
  // discovery nudges via a `kind` discriminator, so the NotificationBell has a
  // single subscription.
  notifications: defineTable({
    userId: v.optional(v.string()),
    guestId: v.optional(v.string()),
    kind: v.union(v.literal("filter_match"), v.literal("discover")),
    savedFilterId: v.optional(v.id("savedFilters")),
    destKey: v.string(),
    destName: v.string(),
    message: v.string(),
    readAt: v.optional(v.number()),
    createdAt: v.number(),
  })
    .index("by_user", ["userId"])
    .index("by_guest", ["guestId"])
    .index("by_filter", ["savedFilterId"]),

  // TRACK D — discovery interactions (save/skip/view event log; learning-ready
  // for Corte 2). sessionId enables anti-repeat + funnel analysis.
  interactions: defineTable({
    userId: v.optional(v.string()),
    guestId: v.optional(v.string()),
    destinationKey: v.string(),
    action: v.union(v.literal("save"), v.literal("skip"), v.literal("view")),
    source: v.union(v.literal("discover"), v.literal("results"), v.literal("feed")),
    sessionId: v.string(),
    createdAt: v.number(),
  })
    .index("by_user", ["userId"])
    .index("by_guest", ["guestId"])
    .index("by_session", ["sessionId"]),

  // TRACK C+E — email + AI need a Clerk-email lookup. Synced via Clerk webhook
  // (user.created/updated). Internal-only; never client-queried.
  users: defineTable({
    userId: v.optional(v.string()),
    email: v.string(),
  }).index("by_userId", ["userId"]),

  // TRACK E — cache LLM "¿y qué hago ahí?" curation per destinationKey so the
  // same card never re-pays for generation. keyed by destinationKey (+ optional
  // vibe hash later). Lives in Convex so both server actions and the client read it.
  curations: defineTable({
    destinationKey: v.string(),
    summary: v.string(),       // short "what to do there" copy
    items: v.array(v.string()), // 3-5 activity bullets
    model: v.string(),
    createdAt: v.number(),
  }).index("by_dest", ["destinationKey"]),
});

Risks & open questions
  • Clearing INITIAL pre-filled arrays (packages, vacation, etc.) changes the first-render appearance of existing live users mid-session — anyone with a bookmark or back-navigation sees chips unselected. Low risk since this is a single-user prototype, but worth noting.
  • The budget filter in computeOptions() is new behavior: setting budget=400 hard-excludes lisbon ($880), cruise ($690), patagonia ($1340), and soft-penalises tulum ($720). With only 6 destinations this could leave 2-3 results or trigger the fallback path. The fallback (return DESTS.slice(0,1)) ensures at least one card is shown, but the Results header saying '3 surprise trips' would need to handle fewer results — Results.tsx:38 already renders `{options.length} surprise trips` dynamically, so this is safe.
  • Deleting Step 3 removes the last user-facing 'review before commit' touchpoint. The fallback is the Results ← button which returns to wizard. Acceptable for this audience but worth documenting in a UI comment.
  • The ±1 grace day in the days filter (d.nights > daysAvailable + 1) is a magic number. If the catalog expands with half-day or 2-night options, revisit the threshold.
  • Should 'Any' be a chip in the days row (explicit selection) or a 'clear' X button that appears only when a value is set? Chip is simpler to implement and visible by default — recommend chip.
  • Does clearing INITIAL.general/weather break any Convex saveDraft call? Check HandoffSheet.tsx confirm() — it passes state fields from usePlanner computed, not raw INITIAL, so clearing arrays should be safe. Verify at src/components/screens/HandoffSheet.tsx before merging.
  • The budget soft-penalty threshold (base > budget * 2 = hard exclude) was chosen conservatively. Should the hard-exclude threshold be tighter, e.g., 1.5×? Depends on whether the user's budget is per-person or total — today budget is per-person (INITIAL:1800) and DEST_BASE is also per-person, so the math is consistent.

      Risks & decisions

      Cross-cutting

      Cross-cutting risks

      • Convex codegen ordering: every frontend import of api.favorites.*/savedFilters.*/notifications.* type-errors until `npx convex dev` regenerates _generated/api. Run convex dev BEFORE editing the frontend for each backend track; this trips up all of B/C/D/E.
      • Scoring is now load-bearing for 4 tracks: if scoreDestinations changes signature (e.g. A adds days/budget), the C cron and D shuffle must pass the new criteria too. Keep criteria a single typed object (FilterCriteria) and thread it everywhere rather than positional args. The savedFilters row already stores daysMax/budget optional for this reason.
      • pinnedKey (B) must be cleared on EVERY re-entry to scored flow (surprise/select/startWizard); missing one strands a user on one destination. Same class of bug as forgetting to reset seen-set in D's discovery loop — both are 'stale session override' hazards.
      • Static 6-DESTS makes C's 'new match' delta almost never fire and makes D cycle in <6 taps. Both are intentional ('design now, swap source at Corte 1') but the solo dev must seed/empty lastMatchKeys and enforce anti-repeat to test. Don't ship D without graceful pool-exhaust reset.
      • Email/AI external deps fail-soft: Resend won't deliver to non-owner until DNS-verified; getEmailByUserId returns null if the Clerk webhook isn't wired; AI Gateway needs AI_GATEWAY_API_KEY. Each must no-op (not crash) when unconfigured so local dev and the affiliate-only MVP still run.
      • Capability-token model is reused widely (guestId, /api/unsubscribe?filterId, opaque Convex IDs). Consistent with existing saveDraft trust (trips.ts:29-31) but means filterId/guestId must never be rendered anywhere else in the UI.
      • Mobile gap: the top-right cluster (page.tsx:91-105) is desktop-only (hidden lg:block). Favorites entry (B), NotificationBell (C/D) need a mobile affordance or they're invisible on the primary MX-market device. Add a mobile top bar when first introducing the bell.

      Key decisions

      • Separate favorites table vs status column on trips: choose SEPARATE — saveDraft's one-draft-per-guest .first() upsert (trips.ts:36-44) and list() (trips.ts:69-80) both assume the current invariant; folding favorites in would force branching every existing trips query. Confirmed by reading both functions.
      • Single scoring source: extract to convex/lib/scoring.ts and re-export to src/lib — NOT two copies. The cron, the hook, and discovery must agree; one module makes the 'identical to old computeOptions' acceptance test trivial.
      • daysAvailable as single number|null (A) vs min/max range: choose single number with null='Any', because the 6 DESTS span nights 3-7 and a range doubles the controls. Revisit when Amadeus variable-length itineraries land.
      • Email transport: Resend via raw fetch (no npm dep) vs resend SDK: choose raw fetch inside a Convex action — minimal footprint for an affiliate-only MVP; add the SDK only if typed helpers become worth 1.4kb.
      • Clerk email access: synced users table via webhook (approach A) vs Clerk Backend API call inside the action (approach B): choose A — cleaner, cacheable, and fails soft (null email = skip send).
      • AI curation caching: cache per destinationKey in a curations table (each destination generates once) vs per-request generation: choose cached — controls cost and latency on the static catalog; key can gain a vibe hash later when curation becomes vibe-aware.
      • Notifications: ONE table with a kind discriminator (filter_match | discover) feeding ONE bell vs per-track tables: choose one — a single reactive subscription, simpler UI, and the bell is built once in Phase 3 and reused by D.
      • Build sequencing: ship A (real scoring + slim wizard) and B (favorites) BEFORE C/D/E — they most directly fix 'feels long' (fewer fields, zero-tap save-and-return) and they harden the scoring + owner-pattern foundations the later tracks depend on.
