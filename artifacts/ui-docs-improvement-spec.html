Executive summary

What to build next, ranked by leverage

The Paradise Plan repo is a working English wizard prototype (landing -> 3-step wizard -> 2.9s fake loader -> top-3 results -> detail -> Expedia handoff) running entirely on 6 hardcoded destinations in src/lib/data.ts, scored synchronously in src/lib/usePlanner.ts. Three contradictory product visions coexist in the repo with no resolution: the BUILT wizard, the 2019 XD checkout flow (docs/flujo-app-ui.md), and the Spanish/Mexico "Drift" discovery-first feed (docs/wiki/*). Your top concern — "feels long" — has a single, verifiable root cause: Wizard Step 1 renders 8 field groups (~1,111-1,142px on a 390px viewport) but computeOptions() (usePlanner.ts:155-166) reads only 3 inputs (general, weather, intensity). Five of the eight collected fields (departure, budget, packages, vacation, travelerTypes) have ZERO effect on results or price. This is both the usability bottleneck AND a trust bug: a $400 budget still surfaces $1,340 Patagonia. The highest-leverage move is to collapse Step 1 to the fields that actually drive output and delete the redundant Step 3 summary — cutting the flow from 3 wizard screens to 1-2 with near-zero engineering risk, since the removed fields are already dead code in the scoring path. The deeper strategic question is whether to keep extending the wizard or pivot toward the documented Drift discovery-first feed; the evidence says keep a lean wizard now as the filter layer and converge toward the feed incrementally, because most of your own product ideas (days filter, optional filters, save, notifications, StumbleUpon discovery, AI experience) align with Drift, not the wizard. Many findings reduce to two clusters: (1) collected-but-unused inputs / fake signals that erode trust, and (2) documentation that describes products you have not built while the built app is barely documented and the README is stale.

⚑ Decide this first

Should the product keep extending the built wizard-first flow, or pivot to the documented Drift discovery-first feed (conversational 'Sorpréndeme' + live flexibility-deal feed) as the primary entry?

RecommendationKeep a drastically slimmed wizard NOW (collapse to 2 mandatory inputs, delete Step 3) to fix the 'feels long' problem cheaply, but explicitly position it as the filter layer of a discovery-first product and start building the Drift feed as the new primary entry surface incrementally. Do not rebuild from scratch and do not keep growing the wizard.
RationaleThe evidence cuts both ways but converges: (1) The wizard is the only thing actually built and it works end-to-end — throwing it away wastes real, functioning code (usePlanner.ts scoring, detail/handoff/Convex persistence). (2) But the wizard is also the source of every top usability finding (8-field Step 1, dead inputs, fake loader) AND it contradicts the stated core differentiator — docs/wiki/arquitectura-mvp.md:12-13 explicitly says 'La interfaz principal no es una caja de búsqueda vacía, sino un feed vivo' and prototipo-drift.md:31 makes 'Sorpréndeme' the main landing entry. (3) Decisively, 6 of your 8 product ideas (days filter, optional filters, save, notifications/emails, StumbleUpon discovery, AI experience) are feed/discovery-shaped, not form-shaped — they cannot be satisfied by a longer wizard. The slimmed wizard is the lowest-risk way to ship usability wins this week while the feed (which needs real data via Amadeus/Travelpayouts) is built behind it. Critically, this requires recording the decision as an ADR so the wizard is treated as a deliberate transitional layer, not abandoned work.
